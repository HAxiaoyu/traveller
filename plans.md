# 旅游规划 Agent（MVP）开发计划文档

**版本**：v1.0  
**文档状态**：已审阅  
**目标**：在最短时间内构建一个可演示、可体验的最小可行性产品，验证“模糊需求→结构化行程+地图可视化”的核心价值。

---

## 1. 项目概述

### 1.1 产品目标
为普通出游人群提供一个 **零门槛** 的 Web 旅游规划助手。用户只需用自然语言说出模糊想法（如“想去日本，喜欢美食和自然”），Agent 即自动完成意图分析、多天路线规划、地图可视化展示与天气提示。

### 1.2 MVP 核心功能
- **自然对话式偏好收集**：多轮引导，自动提取目的地、天数、兴趣标签、体力等。
- **整体路线生成**：输出带城市/景点顺序、每日活动建议、交通方式、大致耗时的行程表。
- **交互式地图展示**：Google Maps 渲染路线和兴趣点标记，支持缩放和详情。
- **天气预报提示**：为每日行程附加目的地天气摘要（不用于自动调整行程）。
- **模型灵活配置**：用户可在设置中自选 LLM 提供商并填入 API Key。
- **会话历史**：左侧栏保留历史规划记录，可回溯、继续对话。

### 1.3 非 MVP 范围（明确不做）
- 预订链接、门票信息
- 精确交通时刻/票价
- 实时天气动态调整行程
- 预算精确计算
- 用户登录/注册
- 多语言（仅中文）

## 2. 用户使用流程（理想路径）

1. 用户打开 Web 应用，自动创建匿名会话。
2. 在聊天框输入：“想去日本，大概7天，喜欢美食和自然风光，不要太累。”
3. Agent 解析并回显确认：“你是想去日本，大约7天，兴趣在美食和自然，体力偏好轻松，对吗？”
4. 用户确认后，Agent 展示“正在规划...”思考过程，流式输出。
5. 规划完成，聊天区展示结构化行程卡片：按天列出地点、活动、交通方式、建议时间。
6. 同时，右侧（或卡片内）嵌入交互式 Google 地图，显示整体路线及每日点标记。
7. 每天行程旁显示一个小天气图标和简短天气描述（小雨，15~22℃）。
8. 用户可在左侧会话列表切换/新建会话，可在设置中更换模型。

## 3. 技术架构总览

| 层级           | 技术选型                                             | 说明                             |
| -------------- | ---------------------------------------------------- | -------------------------------- |
| **前端**       | React + Vite + Tailwind CSS + shadcn/ui              | 实现类 Claude 风格聊天界面       |
| **地图**       | @react-google-maps/api（Google Maps JavaScript API） | 交互地图，路线绘制               |
| **后端框架**   | Python FastAPI                                       | 提供 REST API 和 SSE 流式        |
| **Agent 框架** | LangGraph                                            | 管理规划多步状态机               |
| **LLM**        | 通过 LangChain 兼容接入，用户自选                    | 默认支持 OpenAI、Anthropic、智谱 |
| **数据库**     | SQLite（本地文件）                                   | 存储会话、行程 JSON              |
| **外部 API**   | Google Geocoding API, OpenWeatherMap                 | 补全坐标、获取天气               |
| **部署**       | 本地 / 单容器（Docker）                              | MVP 阶段简单即可                 |

**架构图（文本描述）**：
```
[浏览器] <--SSE/HTTP--> [FastAPI Server] 
                             ├── LangGraph Agent (StateGraph)
                             ├── SQLite 会话存储
                             ├── Google Geocoding API
                             └── OpenWeatherMap API
```

## 4. 前端设计要点

### 4.1 页面布局（单页多面板）
- **左侧栏**（宽约 260px）：会话历史列表，支持新建、切换、删除。
- **主聊天区**：消息列表（用户/AI），支持 Markdown 渲染和行程卡片组件。
- **地图面板**：默认在行程卡片右侧或下方展开，可收起。行程卡片内包含“在地图中查看”按钮。

### 4.2 行程卡片设计
解析后行程 JSON 渲染为时间轴样式：
```
Day 1｜东京
  🕘 09:00 浅草寺（建议 1.5h）
  🕚 11:00 筑地市场（午餐）
  🕐 13:30 秋叶原（自由探索）
  🚆 交通：JR山手线，约 30 分钟
  🌦️ 多云 12~18℃
```

### 4.3 地图交互
- 初始视图自动适配所有天行程的边界（`fitBounds`）。
- 用不同颜色线条区分不同天路线。
- 点击 Marker 弹出 InfoWindow：地点名、建议时段、天气（如适用）。
- 与行程卡片联动：悬停卡片某日，地图高亮当天的点和连线。

### 4.4 设置弹窗
- 模型提供商下拉：OpenAI / Anthropic / 智谱（自定义）
- API Key 输入框（密码遮蔽）
- Google Maps Key 输入框（前端地图使用）
- 天气 API Key 输入框（可选，有默认 key 则可不填）
- 所有 Key 保存至 localStorage，随请求头发给后端（LLM/天气 Key）或前端直接使用（Maps Key）。

### 4.5 流式输出
使用 Server-Sent Events (SSE) 展示“意图分析中...”、“生成行程中...”等过程消息，增强感知。

## 5. 后端 & Agent 详细设计

### 5.1 LangGraph 状态图（核心）

```
State: {
  messages: list,            // 对话历史
  slots: dict,               // 用户画像槽位
  travel_plan: dict,         // 最终行程 JSON
  intermediate_steps: list   // 过程日志
}
```

**节点流转**：
```
START 
  → intent_analysis (循环子图)
  → plan_generation
  → enrichment (并行 geocoding + weather)
  → format_response
END
```

### 5.2 子图：意图分析
内部节点：
- `check_slots`：检查必填项（目的地、天数、兴趣标签）是否完整。
- `generate_question`：若缺失，生成一个自然的追问（不列出选项）。
- `extract_slots`：分析用户最新回复，更新槽位。
- 循环条件：`if all_slots_filled -> exit; else -> generate_question`。

### 5.3 行程生成 Prompt 策略
系统提示词核心约束：
- 角色：资深旅行规划师。
- 输出格式：严格 JSON，包含 `days` 数组，每天含 `city`、`activities`（含名称、坐标、建议时长）、`transport`（方式、估算时长）、`notes`。
- 约束：每天车程≤3小时；项目数量匹配体力等级；符合兴趣；时间安排合理。
- 知识截止警告：不编造坐标，坐标可留空，后补。

### 5.4 工具（LangChain Tools）
1. `get_coordinates(place_name)` → {lat, lng} （调用 Google Geocoding）
2. `get_weather(lat, lng, date)` → {description, temp_min, temp_max} （调用 OpenWeatherMap）

行程生成后，`enrichment` 节点遍历所有活动，并行调用 `get_coordinates` 补充缺失坐标，并对每天城市调用 `get_weather`。

### 5.5 API 设计

| 端点                     | 方法       | 说明                          |
| ------------------------ | ---------- | ----------------------------- |
| `/api/session`           | POST       | 创建新会话，返回 session_id   |
| `/api/chat/{session_id}` | POST (SSE) | 发送用户消息，返回流式事件    |
| `/api/history`           | GET        | 获取会话列表                  |
| `/api/session/{id}`      | GET        | 获取单个会话详情（消息+行程） |
| `/api/session/{id}`      | DELETE     | 删除会话                      |

SSE 事件类型：
- `status`：过程文本（“正在分析您的偏好...”）
- `plan_chunk`：行程 JSON 分块流式传输
- `done`：完成标记，携带完整行程 JSON
- `error`：错误信息

### 5.6 模型配置传递
前端请求头携带：
- `X-Model-Provider: openai`
- `X-Model-Name: gpt-4o`
- `X-Api-Key: sk-...`

后端解析后动态创建 LLM 实例，不持久化。

## 6. 数据库设计（极简）

**sessions 表**
| 字段        | 类型        | 说明           |
| ----------- | ----------- | -------------- |
| id          | UUID        | 主键           |
| created_at  | datetime    |                |
| updated_at  | datetime    |                |
| messages    | JSON (text) | 完整对话历史   |
| slots       | JSON        | 提取的用户偏好 |
| travel_plan | JSON        | 最终行程       |

历史列表查询只需 `id, created_at, title`（title 由第一条用户输入截取生成）。

## 7. 开发阶段与里程碑

### 阶段 1：基础骨架（1-2 天）
- 搭建 FastAPI 项目，集成 LangGraph 最小图（echo）。
- 实现会话创建、消息存储、SQLite 初始化。
- 前端搭建聊天界面骨架，可发送消息、流式接收回显。

### 阶段 2：意图分析（2-3 天）
- 实现 `intent_analysis` 子图，包含槽位检查与自然追问生成。
- 前端展示分析过程文本。
- 测试多种模糊输入。

### 阶段 3：行程生成（2-3 天）
- 编写行程生成 Prompt，与 LLM 联调至稳定输出有效 JSON。
- 实现 `plan_generation` 节点，解析并校验 JSON。
- 前端解析行程 JSON 并渲染时间轴卡片（无地图）。

### 阶段 4：地图与富化（2-3 天）
- 集成 Google Geocoding 补全坐标。
- 集成 OpenWeatherMap 获取天气。
- 前端加载 Google Maps，绘制路线和标记，联动卡片。

### 阶段 5：设置与会话管理（1-2 天）
- 实现设置弹窗，模型/Key 配置。
- 左侧栏会话历史列表，切换/删除。
- 错误处理与降级展示。

### 阶段 6：整体联调与优化（2 天）
- 端到端测试、异常场景处理（无效 Key、LLM 失败、无坐标）。
- UI 细节调整，响应式微调。
- 编写简要操作指引。

## 8. 关键风险与缓解

| 风险                           | 缓解措施                                                     |
| ------------------------------ | ------------------------------------------------------------ |
| LLM 输出 JSON 不稳定           | 使用 JSON 模式或严格 Prompt + 重试；解析失败时回退降级展示文本 |
| 坐标大量缺失导致地图无法绘制   | 必须用 Geocoding 补全，仍失败则跳过该点标记，仅展示文本      |
| Google Maps API 在国内访问受限 | 前端提供备用地图瓦片（如高德），但 MVP 暂不要求；海外用户正常 |
| 流式传输中途断连               | 定时发送 heartbeat 事件，前端自动重连                        |

## 9. 验收标准（MVP Done）

- [ ] 用户可通过自然语言描述景点/天数/偏好完成行程规划。
- [ ] 系统通过多轮对话收集到必要偏好并生成每一天的具体项目。
- [ ] 行程结果以清晰的时间轴卡片展示，并内嵌交互式 Google 地图。
- [ ] 每天的天气预报正确显示。
- [ ] 用户可在设置中更换模型与 API Key，重新规划即刻生效。
- [ ] 会话历史可查看、切换、删除。
- [ ] 无严重崩溃，基本异常有友好提示。
